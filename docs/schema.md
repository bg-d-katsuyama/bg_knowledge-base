本ドキュメントは `docs/architecture.md` セクション3を抽出・拡張したものです。Notion上に手動で構築する5データベースの定義を記載します。

| DB名 | 役割 | 想定行数（5年後） |
|---|---|---|
| ナレッジエントリDB | メイン。全エントリの中央集約 | 数千〜1万 |

## 1. ナレッジエントリDB（メイン）

| プロパティ名 | 型 | 必須 | 説明 |
|---|---|---|---|
| タイトル | Title | ○ | 元メモのタイトル or Claude生成 |
| 日時 | Date | ○ | 発言・作成の発生日時 |
| ソース種別 | Select | ○ | Meet議事録／手動メモ／Slack／Driveファイル／その他 |
| ソースURL | URL | ○ | 元データへの直接リンク |
| 作成者 | Relation → 人DB | ○ | BG内メンバー（誰が記録したか） |
| 関係者（人） | Relation → 人DB（複数）| | 発言者・参加者 |
| 関係先（組織） | Relation → 企業DB（複数）| | 取引先・専門家所属先 |
| 関連プロジェクト | Relation → プロジェクトDB（複数）| | |
| 関連フォルダ/ファイル | URL（複数可） | | Google Drive等のリンク |
| 内容タグ | Relation → タグDB（複数）| ○ | タグ階層を管理するためRelation化 |
| 要約 | Rich Text | ○ | 3〜5行のClaude生成要約 |
| 本文（補完済み） | Rich Text | ○ | 主述補完後のメイン本文 |
| 本文（原文） | Rich Text | ○ | 改変前データの保持 |
| 背景・理由 | Rich Text | | 判断根拠（例：pH7以下とした理由） |
| 処理ステータス | Select | ○ | 未処理／AI加工済／人検証済 |
| 信頼度 | Select | | 高／中／低 |
| 外部キー | Rich Text | ○ | `source_uri + source_timestamp` のSHA256（重複排除用） |
| 作成日時（システム）| Created time | 自動 | |
| 更新日時（システム）| Last edited time | 自動 | |

### Select選択肢

## 構築手順（Phase 0 で手動作業）

1. Notion上に以下の順でDBを作成（依存関係を考慮）
   1. 企業・団体DB
   2. 人DB（企業DBへのRelationを設定）
   3. プロジェクト/テーマDB（人DBへのRelationを設定）
   4. タグDB
   5. ナレッジエントリDB（上記すべてへのRelationを設定）
2. 各DBのIDを取得し、`.env` の以下に設定
   - `NOTION_DB_KNOWLEDGE_ENTRY`
   - `NOTION_DB_PEOPLE`
   - `NOTION_DB_ORGANIZATION`
   - `NOTION_DB_PROJECT`
   - `NOTION_DB_TAG`
3. Notion Integration を作成し、各DBを共有
4. Integration Token を `NOTION_API_TOKEN` に設定
5. 本番環境では Secret Manager に同じ値を登録

## 設計上の注意

- **タグをRelationにした理由**: Multi-selectでは階層構造が作れず、「土壌/堆肥」「土壌/pH」のような分類ができないため
- **本文を「原文」と「補完済み」で二重化した理由**: Claude加工時のニュアンス欠落リスクを緩和（打ち合わせで久保田様が指摘された懸念点）
- **処理ステータスを持たせた理由**: 人による検証フローを組めるようにするため。初期は勝山様・責任者が検証、将来的にスタッフ展開
- **外部キーを保持する理由**: 同じソースを再処理した際の重複作成を防ぐため
