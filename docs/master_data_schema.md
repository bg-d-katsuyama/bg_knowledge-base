## 2. 1 行の粒度（3 案）

### 案 A: 1 行 = 1 KB エントリ（フラット展開）

**説明**: 既存 KB DB をそのまま Sheets に展開する。Notion の表ビューと同等。

**メリット**:
- 実装が最も単純（Claude API 加工不要、KB 取得 → Sheets 書込のみ）
- 原文への遡及がしやすい（1 行 = 1 ソース）
- コスト最小（Claude API ほぼ使わない、Notion + Sheets API のみ）
- 取り込み所要時間が短い（数十分）

**デメリット**:
- **1 行に複数知見が混在**してしまう（例: 1 時間の MTG メモから 10 個の知見が出ても全部 1 行に圧縮される）
- 検索性が低い（知見単位でフィルタできない）
- 久保田様要望の「マスターデータ感」は弱い（実態は KB の鏡）

**想定行数**: 1,099 行

### 案 B: 1 行 = 1 知見/事実（Claude による抽出）

**説明**: 各 KB エントリから Claude が「土壌 R&D に関する個別の知見」を抽出する。1 KB エントリから N 行（0〜10 程度）が生成される。

**メリット**:
- マスターデータとして検索性・引用性が高い
- 1 知見 = 1 行なのでフィルタ・ピボット集計しやすい
- 「白色腐朽菌は ×× に効く」のような事実ベースの行になる

**デメリット**:
- Claude API コスト（Sonnet 4.6 で抽出、推定 **$15〜40**）
- 実装複雑（プロンプト設計、出典トレーサビリティ、重複検出）
- 「知見」の定義が抽象的でブレが出やすい（プロンプト調整が必要）

**想定行数**: 推定 3,000〜8,000 行（1 KB エントリ平均 3〜7 知見）

### 案 C: 2 シート分割（知見マスター + 出典マスター）

**説明**: 案 B の知見シートに加えて、出典シート（1 行 = 1 KB エントリ）を別シートで保持し、知見シートから FK で参照する。

**メリット**:
- スプレッドシート上で正規化された構造
- 知見の元になった会議・ファイルを 1 クリックで辿れる
- 案 B のメリットを保ちつつ、出典トレースが強い

**デメリット**:
- シート 2 枚を同期する複雑性（更新時の不整合リスク）
- 利用者の理解負荷（FK 参照を意識する必要）
- 実装が最も複雑

**想定行数**: 知見シート 3,000〜8,000 行 + 出典シート 1,099 行

---

## 3. 推奨：案 B（1 行 = 1 知見）

**理由**:
- 久保田様の要望「マスターデータ」という表現が、知見の構造化を強く示唆
- 検索・絞り込み・引用が容易（実用面で最大価値）
- 案 C と比べて実装・運用がシンプル

**ただし**:
- 出典トレーサビリティは「出典 URL」列（複数 URL を改行区切り or カンマ区切り）で確保
- 知見定義のプロンプトは初期版を作って、久保田様の検証で調整

## 4. 列構成案（案 B 想定）

| 列 | 型 | 例 | 説明 |
|---|---|---|---|
| 知見ID | 文字列 | `K-00001` | 一意ID（連番 or SHA） |
| 知見タイトル | 文字列 | 白色腐朽菌のリグニン分解能 | 30 字程度の見出し |
| 内容 | 長文 | 白色腐朽菌（Phanerochaete chrysosporium 等）は… | 100〜500 字程度の本文 |
| 領域分類 | 選択 | 微生物 / ミネラル / 堆肥 / LCA / pH / モニタリング / その他 | 大カテゴリ |
| 関連タグ | 文字列 | 白色腐朽菌, リグニン, 堆肥化 | カンマ区切り |
| 発言者・出所 | 文字列 | 市橋氏（東北大）, 田中氏（BG） | 知見の根拠となる人物 |
| 所属 | 文字列 | 東北大学, BG | 発言者の所属 |
| 発生日 | 日付 | 2025-04-10 | 元 KB エントリの発生日 |
| 関連プロジェクト | 文字列 | 土壌形成資材開発 | 該当する場合 |
| 信頼度 | 選択 | 高 / 中 / 低 | Claude による初期判定 |
| 検証ステータス | 選択 | 仮説 / 根拠あり / 実証済み / 要追加検証 | 久保田様が手動で更新可 |
| 出典 KB ID | 文字列 | `kb-2025-04-10-soil-hearing` | 元 KB エントリの page_id |
| 出典 URL | URL | https://notion.so/… | 元 KB エントリの Notion URL |
| 補足メモ | 長文 | （任意） | 利用者が後から書き足せる欄 |
| 作成日時 | 日時 | 2026-05-28 19:30 | 行の作成日時 |
| 最終更新 | 日時 | 2026-05-28 19:30 | 行の最終更新日時 |
| 抽出元 KB エントリ更新日 | 日時 | 2026-05-12 | 再抽出判定用 |

**列構成のポイント**:
- 「領域分類」は事前定義した固定リスト（プルダウン）にする
- 「検証ステータス」は久保田様の手動更新を想定
- 「補足メモ」は人手で書き加えられる空欄

## 5. 抽出ロジック（案 B 想定）

```
1. KB DB から土壌関連タグ付きエントリ 1,099 件を取得
2. 各エントリの本文（補完済み）を Claude API (Sonnet 4.6) に投入
3. プロンプト:
   「以下の会議メモ/ドキュメントから、土壌 R&D に関する個別の知見を
    JSON で抽出してください。1 知見 = 1 オブジェクト。
    領域分類は次のいずれか: 微生物 / ミネラル / 堆肥 / LCA / pH / モニタリング / その他」
4. JSON を行に展開し、Sheets に書き込み
5. 既存知見との重複検出（タイトル類似度 or SHA-based external key）
```

**コスト見積**:
- 1 KB エントリあたり 入力 ~1500 トークン + 出力 ~500 トークン
- Sonnet 4.6 では $3/M input、$15/M output
- 1,099 件 × (1500 × $3 + 500 × $15) / 1M = **約 $13**
- プロンプトキャッシング適用で $10 以下に収まる見込み

## 6. Sheets 書き込みの実装方針

- **UPSERT**: 知見ID（または出典 KB ID + 知見タイトルの SHA）をキーに、既存行は更新、新規は追加
- **論理削除**: KB 側で archive されたエントリ由来の行は「廃止」列を立てて残す（物理削除しない）
- **書き込み単位**: バッチ書き込み API（`spreadsheets.values.batchUpdate`）を使い、API コール数を最小化
- **並行性**: 手動実行 1 つのみ想定。並行実行は想定しない
