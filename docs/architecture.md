## 2. システム全体像

### 2.1 目的

散在する会議メモ・議事録・Drive文書・Slack発言を一元化し、以下を実現します。

- **「誰がいつ何を言ったか」が検索可能**な構造化データベース
- **初見でも文脈が理解できる**ようAIが主述補完・要約した本文
- **ソースURLと原文が保持**された、監査可能な形
- **Slack経由で責任者が指示**、自動で取り込み・更新が走る仕組み
- **スタッフはNotion上で日常的に検索・参照**できるUX

### 2.2 アーキテクチャ概要

```
┌─────────────── データソース層 ───────────────┐
│  Notion既存メモ │ Google Drive │ Google Meet │ Slack │
│                 │ (共有ドライブ)│ (文字起こし)│(特定Ch)│
└─────────┬───────────┬──────────────┬──────────┬──────┘
          │           │              │          │
          ▼           ▼              ▼          ▼
┌─────────────── 取り込み層 (GCP Cloud Run Jobs) ────────────┐
│  各ソースリーダー → 差分検出 → 生データ保存（GCS）         │
└─────────────────────────┬──────────────────────────────────┘
                          ▼
┌─────────────── 加工層 (Claude API) ─────────────────────────┐
│  要約 / タグ付け / エンティティ抽出 / 主述補完             │
└─────────────────────────┬──────────────────────────────────┘
                          ▼
┌─────────────── ナレッジベース層 (Notion) ────────────────┐
│  ナレッジエントリDB（メイン）                             │
│  └ 人DB / 企業DB / プロジェクトDB / タグ階層DB（サブ）   │
└──────────┬────────────────────────────┬──────────────────┘
           │                            │
           ▼                            ▼
┌──── 利用（スタッフ）────┐     ┌──── 制御（責任者）─────┐
│  Notion標準検索         │     │  Slack Bot経由で        │
│  （T1によってはAI追加） │     │  Claude Codeに指示      │
└─────────────────────────┘     └─────────────────────────┘
```

### 2.3 技術スタック

| 層 | 採用技術 | 理由 |
|---|---|---|
| 開発言語 | Python 3.12 | Google API・Notion・Anthropic SDKの公式対応が揃う |
| パッケージ管理 | uv（Astral製） | pipより10〜100倍高速、Claude Codeとの親和性高 |
| リンター | ruff | uvと同じエコシステム、高速 |
| テスト | pytest | 標準 |
| ホスティング | Cloud Run Jobs（バッチ）+ Cloud Run（Slack Bot常駐） | GCP統合、最小運用負荷 |

| シークレット管理 | Secret Manager（GCP） | APIキー・トークンを一元管理 |

## 3. Notionスキーマ設計

### 3.1 データベース構成

全5データベースを作成します。各DBはNotion上で相互にRelationで接続されます。

#### （1）ナレッジエントリDB（メイン）

| プロパティ名 | 型 | 必須 | 説明 |
|---|---|---|---|
| タイトル | Title | ○ | 元メモのタイトル or Claude生成 |
| 日時 | Date | ○ | 発言・作成の発生日時 |
| ソース種別 | Select | ○ | Meet議事録／手動メモ／Slack／Driveファイル／その他 |
| ソースURL | URL | ○ | 元データへの直接リンク |
| 作成者 | Relation → 人DB | ○ | BG内メンバー（誰が記録したか） |
| 関係者（人） | Relation → 人DB（複数）| | 発言者・参加者 |
| 関係先（組織） | Relation → 企業DB（複数）| | 取引先・専門家所属先 |
| 関連プロジェクト | Relation → プロジェクトDB（複数）| | |
| 関連フォルダ/ファイル | URL（複数可） | | Google Drive等のリンク |
| 内容タグ | Relation → タグDB（複数）| ○ | タグ階層を管理するためRelation化 |
| 要約 | Rich Text | ○ | 3〜5行のClaude生成要約 |
| 本文（補完済み） | Rich Text | ○ | 主述補完後のメイン本文 |
| 本文（原文） | Rich Text | ○ | 改変前データの保持 |
| 背景・理由 | Rich Text | | 判断根拠（例：pH7以下とした理由） |
| 処理ステータス | Select | ○ | 未処理／AI加工済／人検証済 |
| 信頼度 | Select | | 高／中／低 |
| 作成日時（システム）| Created time | 自動 | |
| 更新日時（システム）| Last edited time | 自動 | |

#### （2）人DB

| プロパティ | 型 | 説明 |
|---|---|---|
| 氏名 | Title | |
| よみがな | Rich Text | 検索性向上のため |
| 所属 | Relation → 企業DB | |
| 専門領域 | Multi-select | 土壌／LCA／クレジット／流通／開示 等 |
| BG内/外 | Select | 内部／外部 |
| 連絡先 | Email/Phone | |
| 備考 | Rich Text | |

#### （3）企業・団体DB

| プロパティ | 型 | 説明 |
|---|---|---|
| 名称 | Title | |
| 業界 | Select | |
| BGとの関係 | Multi-select | 取引先／研究パートナー／専門家所属先 等 |
| 連絡窓口 | Relation → 人DB | |

#### （4）プロジェクト/テーマDB

| プロパティ | 型 | 説明 |
|---|---|---|
| プロジェクト名 | Title | |
| ステータス | Select | 計画中／進行中／完了／保留 |
| 主担当 | Relation → 人DB | |
| 親テーマ | Relation → 自DB（self） | 階層構造を表現 |

#### （5）タグDB（内容タグ）

| プロパティ | 型 | 説明 |
|---|---|---|
| タグ名 | Title | |
| カテゴリ | Select | 技術／戦略／運用／顧客／その他 |
| 親タグ | Relation → 自DB（self） | タグの階層化 |
| 説明 | Rich Text | タグの意味・用途 |

### 3.2 スキーマ選定の理由

- **タグをRelationにした理由**: Multi-selectでは階層構造が作れず、「土壌/堆肥」「土壌/pH」のような分類ができないため
- **本文を「原文」と「補完済み」で二重化した理由**: Claude加工時のニュアンス欠落リスク（打ち合わせでも勝山様が懸念されていた点）を緩和
- **処理ステータスを持たせた理由**: 人による検証フローを組めるようにするため（初期は勝山様・責任者が検証、将来的にスタッフ展開）

## 4. データ取り込みパイプライン

### 4.1 Notion既存メモ同期（Phase 1）

**トリガー**: 初回バッチ実行、以降は日次（02:00 JST）

**処理フロー**:
1. Notion APIで既存ミーティングメモDBを読み取り
2. 各ページのIDと最終更新日時を記録（`processed_log` テーブル in GCS）
3. 未処理／更新ありのページのみClaude APIへ
4. Claudeに：要約／タグ候補／エンティティ抽出／主述補完を依頼
5. 人DB・企業DB・タグDBへのUPSERT（既存があれば紐付け、新規なら作成）
6. ナレッジエントリDBへ新規行を作成、処理ステータス「AI加工済」で保存

### 4.2 Google Drive / Meet同期（Phase 2）

**Meet議事録**:
- Google Workspaceの設定でMeet録画・文字起こしが所定フォルダに自動保存される前提（T6で確認）
- サービスアカウントで当該フォルダを監視（`files.list` をmodifiedTimeフィルタで毎時実行）
- 新規ファイルを検出 → Drive APIでテキスト取得 → Claude加工 → Notion投入

**手動アップロード議事録・メモ**:
- 指定された共有ドライブ配下（T5で確認）の `.docx` / `.gdoc` / `.md` / `.txt` を対象
- ファイル名に日付パターン（`YYYYMMDD_*`）を推奨するルールを運用で定める

### 4.3 Slack同期（Phase 3）

- Slack Events APIで特定チャンネル（T2・T3で確定）の`message`イベントを購読
- メッセージ単体では短すぎるため、**スレッド単位で1エントリ化**
- スレッドが1時間更新されなかった時点で確定処理、Claude加工 → Notion投入
- ファイル添付がある場合は添付メタ情報をNotionの「関連ファイル」欄に記録

### 4.4 重複排除とUPSERTロジック

- エントリごとに `source_uri + source_timestamp` のSHA256ハッシュを「外部キー」としてNotionに保存
- 再処理時は外部キーで既存エントリを検索 → あれば更新、なければ新規作成
- 人・企業名の同名別人問題は、**初期は同一視して紐付け、ダッシュボードで人間が分離作業**する運用

## 5. Claude API 利用方針

### 5.1 モデル使い分け

| 処理 | 採用モデル | 理由 |
|---|---|---|
| 議事録要約・主述補完 | Sonnet 4.6 | 品質優先、日本語の文脈把握が必要 |
| エンティティ抽出（人・企業） | Haiku 4.5 | 定型処理、コスト優先 |
| タグ候補生成 | Haiku 4.5 | 分類タスク、軽量で十分 |
| Slackメッセージ分類 | Haiku 4.5 | 大量・軽量処理 |
| 検索クエリ拡張（将来） | Sonnet 4.6 | 意味理解が必要 |

### 5.2 コスト最適化策

- **プロンプトキャッシング**: システムプロンプトと「BGの基本方針ドキュメント」を90%割引でキャッシュ
- **Batch API**: リアルタイム性不要な日次処理は50%割引のBatch APIで実行
- **1件あたり上限トークン**: 安全策として出力2,000トークン上限を設定

### 5.3 想定月額

- 通常運用時: **$15〜50/月**（円換算: 約2,300〜7,500円）
- 初期一括取り込み（1回限り）: $80〜200

### 5.4 機微情報の扱い（T8連動）

- Claude APIへの送信データは、Anthropicのデフォルト設定で学習利用されない（エンタープライズ規約）が、念のため以下をルール化：
  - 個人情報（氏名以外の個人属性）をマスキングしてから送信
  - 秘密保持契約対象の詳細条件は送信前にチェッカーで検出・除外
- T8の回答次第で、追加のマスキング処理を実装

## 6. Slack → Claude Code 起動の仕組み

### 6.1 全体フロー

```
[責任者] Slackで /kb-sync コマンド実行
   ↓
[Slack] Events APIがCloud Runのエンドポイントへリクエスト
   ↓
[Cloud Run Bot] コマンド内容を検証・パース
   ↓
[Cloud Run Bot] Cloud Run Jobを起動（Claude Codeを含むコンテナ）
   ↓
[Claude Code (非対話モード)] 指示に従ってタスク実行
   ↓
[Claude Code] 完了をSlackへ通知（DM or スレッド返信）
```

### 6.2 Claude Code の実行方式

**重要な設計判断**: Claude Codeは**Claude APIを利用する非対話モード（`claude --print`）**で起動します。

- 理由1: MAXサブスクリプションの自動化用途は規約違反リスクあり（先の議論参照）
- 理由2: APIキー利用であれば、Cloud Runコンテナから安定的に呼び出せる
- 理由3: 責任者のMAXサブスクリプションは、**責任者ご自身の対話的利用**（壁打ち・確認作業）に温存できる

### 6.3 対応するSlackコマンド（初期案）

| コマンド | 動作 |
|---|---|
| `/kb-sync notion` | Notion既存メモの同期を即時実行 |
| `/kb-sync drive` | Google Drive同期を即時実行 |
| `/kb-sync all` | 全ソースの同期を実行 |
| `/kb-status` | 最終同期日時・処理件数を返す |
| `/kb-rebuild <entry_id>` | 特定エントリの再処理 |
| `/kb-search <query>` | 簡易検索（将来拡張） |

### 6.4 実行権限

- コマンド実行可能なSlackユーザーを**Slackユーザーグループ `@kb-admins`** に限定
- 初期メンバー: 責任者、勝山様
- 他ユーザーが叩いた場合は「権限がありません」と返すのみ

## 7. 権限管理設計

### 7.1 権限レイヤーのサマリー

| レイヤー | スタッフ（閲覧者） | スタッフ（編集者） | 責任者 | 勝山（開発者） |
|---|---|---|---|---|
| Notionナレッジ閲覧 | ○ | ○ | ○ | ○ |
| Notionエントリ追加 | × | ○ | ○ | ○ |
| Notionスキーマ変更 | × | × | ○ | ○ |
| Slack `/kb-*` コマンド | × | × | ○ | ○ |
| GCPコンソールアクセス | × | × | △（閲覧） | ○ |
| GitHubリポジトリ（コード） | × | × | Read | Admin |
| Secret Manager | × | × | × | ○ |

### 7.2 退職・異動時の対応フロー

- 対象者のNotion権限を削除
- 対象者がSlackコマンド権限保有者の場合、`@kb-admins` から除外
- GitHubアクセスを剥奪
- Secret Managerは影響なし（サービスアカウント運用のため）

### 7.3 セキュリティ方針

- すべてのAPIキー・トークンは**Secret Manager**に格納、コード中は環境変数経由でのみ参照
- リポジトリに`.env`ファイルは絶対に含めない（`.env.example`のみ）
- mainブランチはプロテクト、PR必須化、コードオーナー承認必須
- サービスアカウントキーは**90日ごとにローテーション**（Cloud KMSで自動化検討）

## 9. 実装フェーズと工程見積

### Phase 0: 環境構築（1〜2週間）
- GCPプロジェクト作成、サービスアカウント発行
- Secret Manager初期設定
- GitHubリポジトリ作成、`CLAUDE.md` 整備
- Notionスキーマ5DB構築（手動）
- ローカル開発環境（uv, ruff, pytest）セットアップ

### Phase 1: Notion内完結（2〜3週間）
- Notion既存メモを新スキーマへ取り込む単発スクリプト
- Claude API加工ロジック（要約・タグ・主述補完）
- 重複排除とUPSERTロジック
- **ここで一度リリースし、現場レビュー** ← 重要

### Phase 2: Google Drive / Meet連携（2〜3週間）
- Driveサービスアカウント連携
- Meet文字起こし定期取得
- 差分検出と定期ジョブ化（Cloud Scheduler）

### Phase 3: Slack連携（2週間）
- Slack Bot構築、Events API購読
- `/kb-*` コマンド実装
- 権限チェック実装

### Phase 4: 運用定着（随時）
- 検索UI強化（T1の結果次第でNotion AI or 自前検索Bot）
- ダッシュボード整備（処理件数・エラー率・コスト）
- 監査ログの定期レビュー体制

**合計想定工数**: 7〜10週間（週5〜10時間を想定）

## 11. リスクと対策

| リスク | 影響 | 対策 |
|---|---|---|
| Claude APIの料金改定 | コスト増 | 四半期ごとに見直し。代替モデル（Haiku増加等）を検討 |
| Notion APIのレート制限 | 同期遅延 | 指数バックオフリトライ、Bulk APIの活用 |
| Meet文字起こしの品質不足 | ナレッジ精度低下 | 固有名詞辞書の整備、重要会議は手動補正フロー |
| サービスアカウント鍵の流出 | セキュリティ重大事故 | Secret Manager管理、90日ローテーション、監査ログ常時監視 |
| 退職者の発生とアカウント棚卸し不備 | 不正アクセスリスク | 四半期ごとの権限棚卸しを`docs/runbook.md`に明記 |
| スタッフがNotionを使わない | ナレッジ蓄積が止まる | Phase 1完了時にデモ・勉強会を実施、運用ルールを定着 |
| Claude APIの日本語トークン効率悪化 | コスト2〜3倍 | プロンプトの英語化を一部検討（ただし日本語の固有名詞は残す） |
